## 1. 작업 범위
1. **KbIndex 코어 키잉(keying)·후보임계 단일원천 추출**: 현 `KbIndex`의 `MaxSubstringKeyLength`(=32)·`TextKeys`/`SplitTerms`/`BoundedSubstrings`(L≤32 substring cap 포함)·`RequiresLinearContainsFallback`(`query.Length > MaxSubstringKeyLength`, 즉 >32 linear-contains 판정)를 **`internal static` 유틸**(예: `KbKeying` 또는 `KbIndex` 내부 static)로 **추출해 catalog index와 clause index가 동일 키잉·동일 >32 임계 판정을 공유**한다. **별도 검색 엔진/중복 알고리즘 금지**(후보발견 단일원천, 점수만 타입별 분리). **기존 `KbIndex.Build`/`FindCandidates(string)`/`DeterministicSignature()` 시그니처·동작 불변**(linear-contains fallback 포함, 기존 `FindCandidates(>32 query).Count == Entries.Count` 단언 보존) — 기존 `KbTests`(linear==index·`DeterministicSignature`·한글 부분일치·catalog 경로·>32 full-catalog fallback) 전부 PASS.
2. **Clause 인덱스 + `KbSearch.SearchClauses(query, userId, maxResults, asOfDate)`**: KB-WP-01 `ClausePackLoader`로 적재한 `IReadOnlyList<RegulationClause>` 위에 ① 공유 키잉으로 후보 발견(검색 필드 = `ClauseText`·`ClauseRef`; **`RequiresLinearContainsFallback`(>32) 임계는 공유하되 fallback 동작은 타입별 — catalog는 현행대로 전체 `Entries` 반환, clause는 전체 clause 선형 스캔으로 후보 생성(>32 질의에서도 `ClauseText` 부분일치 recall 보존), 동일 반환 shape 강제 아님**), ② clause-type 점수(부분일치 비교는 catalog `Score`와 동일하게 `Contains(..., StringComparison.OrdinalIgnoreCase)`, KbSearch.cs:287 — locale 무관), ③ **`OrderByDescending(Score).ThenBy(ClauseId, StringComparer.Ordinal)`** 안정 정렬, ④ `Take(Math.Max(1,maxResults))`. `ClausePackLoadResult.UsedFallback=true`(Pack 미적재/거부) → **검색 결과 0건 + warning + catalog-only 안내**(예외 0, fail-closed). 생성자/조립은 기존 `KbSearch`(catalog) 경로를 깨지 않게 **additive**(별도 ctor 인자 또는 옵션 Pack/이미-로드된 clause 주입; 미주입 시 빈 Pack → 0건).
3. **신규 `KbClauseSearchResult`(record)** — caller가 추가 lookup 없이 인용/검증 가능하도록 **clause 식별 + 인용 metadata + disclosure**를 모두 포함:
   - clause: `ClauseId`(=`ChunkId`)·`SourceId`·`ClauseRef`·`Snippet`·`Score`.
   - 인용(연결 catalog entry에서): `DocumentName`(Title)·`Version`·`EffectiveDate`·`RepealDate`·`SourceLocator`(Source)·`Category`·`SourceOrg`.
   - 게이트/문구: `Disclosure`(`KbDisclosure`)·`DisclosureReason`·`SnippetAllowed`(bool)·`SearchDate`·`ReviewDraftNotice`(="검토용 초안").
   - clause↔catalog 조인은 **`SourceId`로 결정적**. **catalog 매칭 실패 시 fail-closed**: `Disclosure=MetadataOnly`/`SnippetAllowed=false`/`Snippet=""` + warning(임의 노출 금지).
4. **신규 `ClauseSnippetAllowed` 게이트(`KbAccessPolicy`)** — `SourceTextAllowed`와 **분리된 별도 게이트**:
   - `public static bool ClauseSnippetAllowed(RegulationCatalogEntry entry)`: **`Disclosure==PublicCited` AND 비-placeholder 인용/승인 metadata**일 때만 `true`. 필수 확인값은 `version`·`effective_date`·`approval_status`·`license_status`이며, 각 값이 blank/`CONFIRM_*`/`NOT_LOADED`이면 fail-closed. `PROD_ONLY`·`MANUAL_APPROVAL_REQUIRED`·`MetadataOnly`·unknown·placeholder → `false`.
   - **단일 원천(중복 도출 금지)**: PublicCited 판정은 기존 `PublicStatuses.Contains(entry.Status.Trim())`(OrdinalIgnoreCase)를, placeholder/blank 판정은 기존 `IsMissingGateMetadata`를 **그대로 호출**한다(공개 status 집합·placeholder 규칙 재구현 금지). `KbAccessDecision.ClauseSnippetAllowed`(아래)에 저장하는 값과 public 정적 메서드 반환값은 **동일 private 헬퍼 1곳**에서 산출해 항상 일치(두 곳 독립 계산 금지 — drift 0).
   - `KbAccessDecision`에 **`bool ClauseSnippetAllowed = false` 를 기본값 있는 positional 파라미터로 add-only**(기존 4-인자 생성 호출부 불변). 기존 `Evaluate(...)`는 이 값을 채워 반환.
   - **`SourceTextAllowed`는 false 불변**(절대 변경 0). ClauseSnippetAllowed=false면 **항상 `Snippet=""`**로 발췌 0. 거부 사유는 `DisclosureReason`/warning에만 둔다(비어 있지 않은 안내문을 `Snippet`에 넣지 않는다).
5. **Snippet 결정성·상한·정규화 (단일 알고리즘 — 분기 금지)**: 발췌는 다음 **단일 결정적 알고리즘**으로만 산출한다(either/or 금지). ① 정규화 = `ClauseText`의 `char.IsControl(ch)`(개행 `\r`·`\n`·`\t` 포함) → 공백 1개 치환 후 연속 공백 단일화 + 양끝 Trim. ② 쿼리 토큰(`SplitTerms`) 중 정규화 문자열에 `IndexOf(term, StringComparison.OrdinalIgnoreCase)`로 **가장 먼저(최소 offset)** 등장하는 위치 `m`(인자없는 `IndexOf`/`ToLower`/CurrentCulture 금지 — locale 무관). **매칭 없음**(ClauseRef-only·>32 fallback 포함) → `m=0`. ③ `Snippet = normalized.Substring(m, Math.Min(32, normalized.Length - m))` — 시작점 `m`에서 **우측으로만** 확장, 좌측 패딩 0, **상한 = 32**(=`MaxSubstringKeyLength` 값 재사용, 의미는 표시용 상한으로 별개·전체 본문 미반환). 비결정 0. 게이트 false면 본문 미접근(`Snippet=""`).
6. **Clause 유효구간(asOf 경계 — inclusivity 고정)**: `EffectiveDate`/`RepealDate`를 KB-WP-01과 동일 결정적 파싱(`yyyy-MM-dd`, `CultureInfo.InvariantCulture`, `DateOnly`)으로 처리. **active 판정 = `EffectiveDate <= asOf AND (RepealDate blank OR asOf < RepealDate)`** — **시행일 당일 active(포함)·폐기일 당일 inactive(제외, 폐기일=효력 종료일)·blank `RepealDate`=무기한 active**. **out-of-window(시행 전 `asOf < EffectiveDate` / 폐기 후 `asOf >= RepealDate`) clause는 후보에서 제외한다**(결과 미포함; `(기준일 외)` 표식 방식은 쓰지 않음) → §1.2의 정렬·`Take`는 **in-window clause만** 대상이라 out-of-window가 top-N 슬롯을 소비하지 않는다. non-empty이면서 파싱 실패한 `EffectiveDate`/`RepealDate`는 warning(필드·source_id 포함) + **해당 clause를 유효구간 밖으로 보아 제외**(ADR-013 §결정5 '미파싱 EffectiveDate=비활성' 확장 — 미파싱 RepealDate도 폐기 간주)·`SnippetAllowed=false`. 전 과정 결정적·테스트 고정.
7. **검색 행위 해시 audit 재사용**: clause 검색도 기존 `TaskLogWriter`/`TaskLogEntry` 스키마(UserId/RequestHash/OutputHash 모두 SHA-256 hex)로 audit, **원문/쿼리 평문 미저장**. task type은 catalog `KbCatalogSearch`와 구분해 `KbClauseSearch`(또는 동등 구분값)로 둔다. `OutputHash`는 결과 metadata 요약을 `LogHash.Sha256Hex`로만 기록(원문/Snippet 평문 디스크 미저장). `auditRuleVersion` 없으면 기존 패턴대로 미기록+warning.
